# Reweave Frontend

React/Vite browser app for Reweave. The UI calls the backend API for persisted
puzzles, profiles, auth-backed edits, and stats, while board search and puzzle
checking run locally through the Rust WASM package in `frontend/pkg/`.

## Routes

- `/`: latest public puzzle summaries.
- `/search`: searchable puzzle summaries with size and given-letter filters.
- `/how-to-play`: tutorial page.
- `/create`: authenticated puzzle creation flow.
- `/play/:puzzleId`: puzzle play page.
- `/profile/:username`: public user profile with created and completed puzzles.

## Local Development

Install dependencies from the repository root:

```sh
pnpm --dir frontend install
```

Build the WASM package before the first Vite build or after Rust/word-list
changes:

```sh
wasm-pack build frontend --target bundler --out-dir pkg
```

Run the frontend dev server:

```sh
pnpm --dir frontend run dev
```

Vite proxies `/api` to `http://localhost:3000`, so run the local backend from
`reweave/` when testing create, play, profile, search, and stats requests.

## Environment

- `VITE_API_URL`: optional API base URL. If unset, the app uses same-origin
  `/api`, which is proxied to the local backend during Vite development.
- `VITE_CLERK_PUBLISHABLE_KEY`: required by `main.tsx` for Clerk auth.

Backend auth and database environment variables are documented in the repository
root README and `AGENTS.md`.

## WASM Contract

`frontend/src/lib.rs` exports small `wasm-bindgen` functions over shared Rust
game logic:

- `find(width, height, letters)`: finds all dictionary words on a candidate
  answer board during puzzle creation.
- `load_puzzle(puzzle)`: loads a persisted puzzle into WASM global state before
  play checks.
- `load_puzzle_for_create(width, height, words)`: loads temporary puzzle state
  after the creator locks a word list.
- `check(letters)`: compares a board against the loaded puzzle and returns
  found, missing, and extra words.

`frontend/pkg/` is generated by `wasm-pack` and should not be edited manually.
The generated TypeScript type for `check` is `any`; page boundaries cast it into
the discriminated frontend `PlayWords` shape.

## Create Flow

Before locking the word list, the board represents the solved answer and calls
`find` to show every dictionary word available on the board. After locking,
`load_puzzle_for_create` stores the chosen word set in WASM, and the board
switches into clue-selection mode.

In clue-selection mode, `hardSet` means “visible to the player at puzzle start.”
Submitting sends:

- `answer`: the solved board string.
- `letters`: the starting board string, with hidden answer letters replaced by
  `_`.
- `words`: the locked target word list.

Size changes are confirmed because they clear the current board. Submission uses
a description modal before calling `POST /api/create`.

## Play Flow

The play page fetches `/api/puzzle/:puzzleId`, calls `load_puzzle`, then checks
each board update through WASM `check`. `puzzleFetched` is tri-state:

- `undefined`: loading.
- `false`: not found or failed.
- `true`: ready to play.

Stats are deduplicated with refs so play and completion events are counted once.
Completion is counted only after user interaction or hint usage, not merely from
loading an already solved board. Revealed tiles become fixed for the current
session, and full-solution reveal marks the player as having given up.

## Frontend Data Shapes

Word lists are discriminated unions:

- `CreateWords`: `{ kind: "create", all }`
- `PlayWords`: `{ kind: "play", found, missing, extra }`

`PuzzleSummary` mirrors the camelCase JSON returned by `/api/puzzles`,
`/api/profile`, and puzzle metadata updates.

## Tooling Notes

- React 19 is used with the React Compiler via `reactCompilerPreset()` in
  `vite.config.ts`. Do not add `useMemo`, `useCallback`, or `memo` only for
  referential stability.
- Aliases are `@` for `src`, `@components` for shared components, and `@wasm`
  for generated WASM output.
- `vite-plugin-wasm` and `vite-plugin-top-level-await` are required for the
  generated WASM package.
